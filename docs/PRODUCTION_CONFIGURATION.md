# Production Configuration

This document summarizes the production controls already present in source and the required operator configuration.

## Docker Compose

`docker-compose.yml` defines:

- `backend` Express API container.
- `frontend` Next.js standalone container.
- `nginx` reverse proxy container.
- Shared Docker network `nati_nest`.
- Persistent volumes for uploads, backups, and logs.
- Health checks for all services.
- CPU and memory limits.
- Read-only container filesystems where possible.
- Dropped Linux capabilities with only the minimum capabilities re-added for Nginx.

## NPM Version Pinning

Both Dockerfiles install `npm@11.6.2` before `npm ci`. This keeps Docker dependency resolution aligned with the lockfiles generated by local npm 11.

## Backend Runtime

Production startup performs:

- Environment validation.
- Prisma database connectivity retry.
- Express middleware setup.
- Socket.IO setup.
- Graceful shutdown on `SIGINT` and `SIGTERM`.

Health endpoints:

- `/health`: verifies API and database connection.
- `/ready`: lightweight readiness response.

## Nginx

`nginx/nginx.conf` provides:

- Reverse proxy to frontend and backend.
- API/auth rate limiting.
- WebSocket support for Socket.IO.
- Upload proxying.
- Static asset caching.
- Security headers.
- Gzip compression.

Add TLS certificates or place the stack behind a managed HTTPS load balancer before production launch.

## Logging

The backend logger honors:

- `LOG_LEVEL`
- `LOG_SLOW_REQUEST_MS`
- `LOG_SLOW_QUERY_MS`

Application logs are mounted to the `app_logs` Docker volume in Compose.

## Monitoring

Supported checks:

- HTTP `GET /health`
- HTTP `GET /ready`
- Docker health checks
- Optional Sentry DSN variables for backend/frontend error tracking

Recommended production monitoring:

- Uptime probe for `/ready`.
- Database latency and availability alerts from `/health`.
- Container restart alerts.
- Disk usage alerts for uploads, logs, and backups.

## Backups

Scripts are available under `scripts/`:

- `backup.ps1`
- `restore.ps1`

Backups should include:

- PostgreSQL database dump.
- Upload volume or Cloudinary account configuration.
- `.env` secret inventory stored securely outside Git.

Run restore drills before owner handover.

## Security Checklist

- Use HTTPS.
- Use strong unique JWT secrets.
- Rotate seed passwords.
- Confirm CORS origins.
- Configure Cloudinary or backup uploads volume.
- Restrict database access to application hosts.
- Keep rate limits enabled.
- Run `npm audit` in both workspaces before release.
